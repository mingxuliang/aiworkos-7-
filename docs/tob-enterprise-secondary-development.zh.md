# AI Work OS / QwenPaw 二次开发与 ToB 企业级改造——详细分析

**文档版本：** 1.0  
**适用范围：** 私有化交付、行业大客户、集团多租户场景  
**说明：** 本文档侧重「从产品现状走向企业级 ToB」的架构与工作拆分，可与既有控制台（Console）、Workspace、安全模块演进对齐。

---

## 一、背景与目标

### 1.1 业务背景

当前产品在对话与工作台场景中已具备 Agent、频道、会话、定时任务、Skills/Tools/MCP、文件工作区等能力。面向个人或小团队时，**以 Agent 维度隔离工作目录**即可运转；但面向 **ToB / 企业客户** 时，采购方通常要求：**身份与权限（IAM）、数据边界、审计追溯、合规与安全基线、运维与可观测性、集成与企业 SSO** 等与「消费级产品默认假设」不一致的能力。

### 1.2 改造目标（建议写成对外 SLA / 对内 OKR）

| 目标域 | 企业客户典型诉求 | 改造方向摘要 |
|--------|------------------|--------------|
| 身份与访问 | 账号生命周期、角色、最小权限 | RBAC/ABAC、策略引擎、与 IdP 集成 |
| 数据与隔离 | 团队/部门/项目数据不可互窥 | 多租户模型 + 存储前缀/卷隔离 + 加密 |
| 文件与知识 | 文件级权限、外发控制 | 统一文件网关、ACL、DLP 钩子 |
| 合规审计 | 谁在何时访问了什么 | 不可篡改审计日志、留存策略、导出 |
| 运维交付 | 私有化、升级、灰度 | 配置体系、健康检查、备份恢复、License |
| 集成 | AD/LDAP、钉钉飞书、工单 SIEM | SCIM、Webhook、事件总线、API 网关 |

## 二、现状差距分析（与二次开发切入点）

### 2.1 已识别的产品能力边界（示例）

在「文件隐私」类需求上，若当前隔离主要在 **Agent + 本地目录**（例如按 `agent_id` 划分 workspace），则企业视角下的缺口通常包括：

1. **缺少面向「自然人 / 组织单元」的文件 IAM**：同一控制台用户、多 Agent、多技能路径下的访问边界不清晰时，难以通过审计回答「谁读了哪份合同」。
2. **工具链绕过风险**：若 MCP / 本地工具可直接访问 OS 路径，则仅靠 UI 隐藏或 Agent 约定无法通过安全测评。
3. **运维侧一致性**：单机 `chmod` 类手段难以在多副本、容器化、网络存储上一致执行，也不利于批量交付。

### 2.2 二次开发的主战场（分层）

| 层级 | 二次开发重点 | 价值 |
|------|----------------|------|
| **策略与网关层** | 所有文件类 API 统一鉴权与路径规范化 | 防绕过、可审计 |
| **租户与组织模型** | Tenant → Org → Project → Workspace | ToB 计费与隔离基石 |
| **控制台体验** | 权限向导、数据分级标签、审批流 | 降低误配置与安全事故 |
| **可观测与合规** | 结构化审计、导出、对接 SIEM | 中标与续约关键项 |

## 三、ToB 企业级参考架构（改造蓝图）

### 3.1 逻辑架构（文字描述）

- **接入层**：企业 SSO（OIDC/SAML）、可选反向代理、mTLS（金融行业常见）。
- **应用层**：控制台 BFF、核心业务 API、异步任务（导入导出、索引构建）。
- **策略层**：Authorization Service（RBAC + 资源级策略 + 可选 ABAC 标签）。
- **数据层**：关系库（用户/租户/权限/审计）、对象存储或统一文件服务、向量与检索（若涉及 RAG）。
- **执行层**：Agent 运行时、Tool/MCP 执行沙箱（进程/容器/权限降权）。
- **运维层**：配置中心、密钥管理（KMS/Vault）、日志与链路追踪、备份。

### 3.2 多租户模型（推荐演进路径）

1. **租户隔离（逻辑）**：所有业务表带 `tenant_id`；对象存储按前缀隔离；缓存键带租户维度。
2. **强隔离（物理）**：大客户专属集群或专属命名空间；密钥与客户 KMS 绑定。
3. **混合**：默认逻辑隔离 + 「白金租户」物理隔离（计费与交付绑定）。

## 四、分阶段二次开发路线图

### 阶段 A：企业化「底座」（0–3 个月，强烈建议优先）

**交付物**

- 租户 / 组织 / 成员模型与控制台管理后台雏形。
- SSO（OIDC）与可选 SAML；会话与令牌轮换策略文档化。
- **统一审计**：登录、权限变更、文件读写（至少高危操作）、Agent 配置变更、导出行为。
- API 网关层：`tenant_id` 注入、速率限制、请求 ID。

**验收口径**

- 可演示「同一租户内角色差异」与「跨租户不可见」（含 API 自动化用例）。
- 审计日志可按用户、资源、时间窗口导出 CSV/JSON。

### 阶段 B：资源级权限与文件网关（3–6 个月）

**交付物**

- 文件服务抽象：列表、读取、写入、删除、打包下载均走同一 Policy Engine。
- ACL 模型：`owner`、`visibility`（私有/团队/组织）、可选「敏感分级」标签。
- MCP/Tool **路径白名单**与动态策略注入（避免开发者工具绕过）。
- 控制台：文件夹权限、继承规则、敏感目录模板。

**验收口径**

- 渗透测试项：路径穿越、符号链接、压缩解压路径逃逸用例全部拦截或有明确告警。
- 任意工具调用链上有同一套鉴权决策 ID（便于审计关联）。

### 阶段 C：合规、运维与商业化封装（6–12 个月）

**交付物**

- 数据留存与删除策略（GDPR/个人信息场景可按模块裁剪）。
- 备份恢复 Runbook + 自动化校验（含租户级恢复演练）。
- License / 用量计量（Token、存储、并发会话、Agent 数）。
- 私有化安装器与健康检查脚本；离线升级包流程。

**验收口径**

- 备份 RPO/RTO 与客户合同对齐；升级文档通过客户运维评审。

## 五、功能模块改造清单（可按 Scrum 拆 Epic）

### 5.1 控制台（Console）

| Epic | 说明 |
|------|------|
| 租户与组织 | 租户开户、子部门、邀请成员、成员状态 |
| 角色权限 | 系统角色 + 自定义角色；权限模板（安全管理员、开发者、只读审计） |
| 文件与 Workspace | 权限继承 UI、敏感标签、外发拦截提示 |
| 安全中心 | SSO 配置、IP 允许列表、会话策略、API Key 生命周期 |
| 运维面板 | 版本、配置快照、日志级别、队列积压 |

### 5.2 后端与中间件

| Epic | 说明 |
|------|------|
| AuthZ 服务 | PDP/PEP 分离；决策缓存；批量鉴权接口 |
| 文件网关 | 规范化路径、病毒扫描钩子（可选）、下载水印（可选） |
| 事件总线 | `resource.accessed`、`policy.denied` 等标准事件 |
| 集成 | SCIM 用户同步、Webhook、审计导出到 Syslog |

### 5.3 Agent / Tool / MCP

| Epic | 说明 |
|------|------|
| 执行隔离 | 最小权限 OS 用户、只读根、临时目录回收 |
| 工具注册 | 工具声明所需权限范围；运行时校验 |
| 配置下发 | Agent 仅能获得当前租户允许的路径与环境变量 |

## 六、安全与合规要点（ToB 售前常见问卷）

1. **身份**：MFA 策略、会话超时、刷新令牌轮转、服务账号与人工账号分离。  
2. **传输与存储**：TLS 版本、静态加密（磁盘/KMS）、密钥轮换。  
3. **审计**：完整性（WORM 或远程只读 SIEM）、时钟同步、管理员操作双人复核（可选）。  
4. **隐私**：最小采集、字段脱敏显示、导出审批流。  
5. **供应链**：依赖 SBOM、CVE 响应 SLA、构建签名与镜像扫描。

## 八、风险与缓解

| 风险 | 缓解 |
|------|------|
| 过度设计导致交付延期 | 先做租户 + SSO + 审计 + 文件网关 MVP |
| 工具链绕过 | 统一 PEP；禁止未注册工具直访磁盘 |
| 数据迁移复杂 | 早期冻结 `tenant_id` 与资源 URI 规范 |
| 客户合规条款冲突 | 模块化售卖（审计增强包、专属 KMS 包） |

---

## 九、结论

将产品从「Agent 级工作目录隔离」推进到 **ToB 企业级应用**，二次开发的核心不是简单增加页面，而是构建 **租户化数据平面 + 统一授权决策 + 全链路审计 + 可运维交付**。建议按阶段 A→B→C 递进：**先做身份与审计底座，再做文件与工具链策略闭环，最后封装合规与商业化运维能力**，以降低架构返工与安全负债。
